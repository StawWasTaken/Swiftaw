## 🔧 What YOU Need to Build (Fortized)

### BEFORE YOU START: Database Changes

```sql
-- Add these columns to your users table
ALTER TABLE users ADD COLUMN (
  cloud_user_id UUID UNIQUE,
  is_linked_to_cloud BOOLEAN DEFAULT FALSE,
  last_login TIMESTAMP
);

-- Create this new table
CREATE TABLE cloud_account_links (
  id BIGINT PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
  cloud_user_id UUID NOT NULL,
  fortized_user_id BIGINT NOT NULL REFERENCES users(id),
  subaccount_id UUID NOT NULL,
  username VARCHAR(255) NOT NULL,
  email VARCHAR(255) NOT NULL,
  linked_at TIMESTAMP DEFAULT NOW(),
  is_active BOOLEAN DEFAULT TRUE,
  created_at TIMESTAMP DEFAULT NOW(),
  
  UNIQUE(cloud_user_id, subaccount_id),
  UNIQUE(subaccount_id),
  FOREIGN KEY(fortized_user_id) REFERENCES users(id)
);
```

## 🔐 7 API Endpoints You Must Build

All endpoints handle multi-account (subaccount) support. Copy the exact specs from `FORTIZED_INTEGRATION.md`.

### Endpoint 1: Cloud Callback (OAuth Return)
```
POST /api/auth/cloud-callback
Purpose: When user logs into Cloud and is sent back to Fortized
Input: JWT token from Cloud
Output: Fortized auth token + account selection (if multiple)
```

### Endpoint 2: Link to Cloud
```
POST /api/accounts/link-to-cloud
Purpose: Create new Fortized account from Cloud dashboard
Input: Cloud user ID, desired username
Output: New subaccount created
```

### Endpoint 3: Get User's Accounts
```
GET /api/accounts/{cloud_user_id}
Purpose: Fetch all Fortized accounts linked to a Cloud user
Output: List of subaccounts with metadata
```

### Endpoint 4: Switch Subaccount
```
POST /api/auth/switch-subaccount
Purpose: User switches to different Fortized identity
Output: New Fortized session token
```

### Endpoint 5: Disconnect Account
```
DELETE /api/accounts/{cloud_user_id}/{subaccount_id}
Purpose: Remove Fortized account from Cloud
Output: Confirmation
```

### Endpoints 6-7: Verification (Utility)
```
GET /api/auth/verify-cloud-token
GET /api/auth/public-key
Purpose: Validate Cloud JWTs, get public key for signing
```

## 🔑 Key Integration Points

### 1. Login Flow

**Before (Traditional):**
```
User → Fortized Login → Email + Password → Fortized DB → Play
```

**After (Cloud-Based):**
```
User → Fortized Login 
  → Click "Sign in with Swiftaw Cloud" 
  → Redirect to Cloud 
  → Cloud Login/Signup 
  → Cloud redirects back with JWT 
  → Fortized validates JWT 
  → Account picker (if multiple) 
  → Play
```

### 2. Subaccount Management

Users can now have multiple Fortized identities under one Cloud account:

```
Cloud Account: alice@example.com
├── Fortized Account 1: @gaming_pro (last played: 2 hours ago)
├── Fortized Account 2: @mod_admin (last played: 1 day ago)
└── Fortized Account 3: @streamers_alt (last played: 1 week ago)
```

From Cloud dashboard, Alice can:
- Switch between accounts instantly
- Manage each account separately
- Disconnect any account anytime

### 3. Security Model

```
Cloud Responsibility:
- User authentication (email/password)
- JWT generation
- 2FA, password management
- Session management

Fortized Responsibility:
- Validate Cloud JWT
- Create/link subaccounts
- Enforce Fortized permissions
- Track last_login in Cloud DB
```

## ⚠️ Common Issues & Solutions

| Issue | Cause | Solution |
|-------|-------|----------|
| "Invalid token" error | JWT validation failed | Check Cloud public key, verify token format |
| Account picker doesn't show | Query returned no accounts | Check `cloud_account_links` table, verify UUID |
| Can't switch accounts | Wrong subaccount_id | Verify ID matches database |
| Login redirect loops | Callback URL mismatch | Check exact redirect URL in Cloud |
| Slow login | JWT validation slow | Cache public key, don't fetch every time |

## ❓ FAQ

**Q: Do we have to use Cloud?**  
A: No, but it's the future of Swiftaw. Integration is encouraged but optional initially.

**Q: What about existing Fortized users?**  
A: They can continue with local login. Cloud linking is optional at first.

**Q: Can users have Cloud + non-Cloud accounts?**  
A: Yes, during transition. Eventually all accounts will be Cloud-linked.

**Q: What's the cost of Cloud integration?**  
A: Zero additional cost. It's built with Supabase free tier + standard infrastructure.

**Q: How do we rollback if something breaks?**  
A: Database rollback + keep local auth working. Users fall back to traditional login.
