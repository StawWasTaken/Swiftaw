Department 02

Safety

Safety reviews every release before it ships. We red-team, audit privacy impact, run threat models, and occasionally block a feature we don't think is ready.

Our pre-release checklist

Eight things we check on every release

✓
Threat model refresh
Does this release change our threat surface? If yes, update and publish the model.
✓
Data-flow review
Track every new piece of user data the feature touches. Where it lives, who can see it, how it's deleted.
✓
Red-team pass
Safety assigns a red-team reviewer unfamiliar with the feature to try to break it.
✓
Privacy impact
A written privacy impact note — published externally if users are affected.
✓
Rollback plan
Every release has a documented rollback, tested at least once before ship.
✓
Abuse considerations
How could this feature be used against someone? We close the most realistic abuse paths before ship.
✓
Accessibility audit
Keyboard, screen reader and color-contrast checks for any UI change.
✓
Incident-report draft
The skeleton of the post-mortem is drafted before ship — so we're ready if we need it.
How a review runs

The Safety process

Intake

R&D hands off a draft design. Safety assigns a lead reviewer.

Review

The reviewer walks the checklist and files blocking or nonblocking notes.

Signoff

Blocking items resolved. A written signoff goes on the release.

Watch

We keep eyes on the release for the first 30 days and publish what we see.

Every "no" we give is a product decision. Safety doesn't exist to sign off — it exists to catch the things we'd regret.
