📊 HOW IT WORKS (Architecture)

CURRENT (Before Cloud):
  User → Fortized Login → Fortized DB → Play

NEW (After Cloud):
  User → Fortized Login
       → Click "Sign in with Swiftaw Cloud"
       → Redirected to cloud.swiftaw.io
       → Cloud auth (or signup)
       → Cloud redirects back to Fortized with JWT token
       → Fortized validates token
       → Account picker (if multiple accounts)
       → Play

RESULT:
  • One Cloud account
  • Up to 10 Fortized subaccounts per user
  • Instant account switching
  • Unified security management

❓ QUICK FAQ

Q: Do we have to do this?
A: Not immediately, but it's the future architecture. Better to integrate now.

Q: Will existing users be affected?
A: Not immediately. Cloud login is optional at first. Gradual migration.

Q: What if something breaks?
A: Keep local auth as fallback. Database rollback available. Zero downtime.

Q: How long will it take?
A: 3 weeks for experienced team. Might be 4 weeks if you're learning as you go.

Q: Do we need to change our current game code?
A: Minimal changes. Login page + account switching UI. Game logic stays same.

Q: Will users need new passwords?
A: Only if they create a new Cloud account. Existing Fortized passwords work.

  ↓ enter email + password
  ↓ fortized database lookup
  ↓ log in directly
  ✓ done

NEW WAY:
  fortized.com/login
  ↓ click "Sign in with Swiftaw Cloud"
  ↓ redirect to cloud.swiftaw.io
  ↓ cloud authentication (Cloud's responsibility)
  ↓ redirect back with JWT token
  ↓ Fortized validates token
  ↓ account selection (if multiple accounts)
  ↓ create session
  ✓ done

  - Unified security (2FA in one place)
